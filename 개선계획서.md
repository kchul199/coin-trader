## 1. 문서 목적

본 문서는 현재 프로젝트의 문서-구현 불일치 사항을 우선적으로 정리하고,
즉시 수정해야 하는 항목과 추후 단계적으로 적용할 개선 사항을 분리하여 관리하기 위한 계획서다.

핵심 원칙은 다음과 같다.

1. 실제 오류가 발생하는 기능을 먼저 수정한다.
2. 사용자 흐름과 문서의 불일치를 해소한다.
3. 기능 확장은 정합화 이후 단계적으로 진행한다.

## 3. 우선 수정 계획

우선 수정은 `P0 -> P1 -> P2` 순서로 진행한다.

### 3.1 P0: 실행 오류 및 핵심 정합성 수정

#### 목표

실제 런타임에서 실패한 기능을 먼저 복구하여, 핵심 사용자 플로우가 중단되지 않도록 한다.

#### 작업 항목

1. 2FA 세션/저장 정합성 수정
   - 대상 파일
     - `backend/app/api/v1/auth.py`
     - `backend/app/dependencies.py`
     - `backend/app/database.py`
   - 내용
     - `2fa/setup`에서 저장한 `totp_secret`가 실제 DB 세션에 반영되도록 수정
     - `2fa/verify` 성공 시 이후 로그인에서 TOTP 코드가 실제로 요구되도록 보장
     - 현재 사용자 객체와 커밋 세션이 분리되는 구조를 점검

2. 긴급 정지 상태 조회 500 수정
   - 대상 파일
     - `backend/app/api/v1/emergency.py`
   - 내용
     - Redis 값이 `str`로 반환되는 경우 `.decode()` 호출로 실패하는 문제 수정
     - 글로벌 정지 플래그와 전략별 정지 플래그 모두 동일한 방식으로 안전 처리

3. 백테스트 작업 큐 라우팅 수정
   - 대상 파일
     - `backend/app/tasks/celery_app.py`
     - `infra/docker-compose.yml`
   - 내용
     - `tasks.run_backtest`가 실제로 worker에서 소비될 수 있도록 큐 라우팅 추가
     - 필요 시 worker 구독 큐에 `backtest` 추가 또는 `maintenance`/`trading` 중 적절한 큐로 라우팅

4. 내장 검증 스크립트 보정
   - 대상 파일
     - `scripts/test_docker.sh`
   - 내용
     - `docker compose ps --format` 결과 해석 로직 수정
     - 현재 Compose 출력에서 `Up`, `healthy`, `running` 등을 정확히 판별하도록 개선

#### 완료 기준

- `POST /api/v1/auth/2fa/setup -> POST /api/v1/auth/2fa/verify -> 로그인 시 TOTP 요구` 확인
- `GET /api/v1/emergency/status`가 200 응답 반환
- `POST /api/v1/backtest/run` 이후 작업 상태가 `running` 또는 `completed/failed`로 진전
- `bash scripts/test_docker.sh` 실행 시 현재 스택 상태를 정확히 판정

## 6. 검증 원칙

각 단계 완료 시 다음 기준으로 검증한다.

1. API 단위 검증
   - HTTP 상태코드
   - 요청/응답 계약
   - 인증/권한 체크

2. 사용자 플로우 검증
   - 브라우저 기준 실제 진입 흐름
   - 문서대로 따라 했을 때 막히지 않는지 확인

3. 작업 큐 검증
   - Celery 태스크가 실제로 소비되는지
