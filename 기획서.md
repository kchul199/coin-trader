## 1. 프로젝트 개요 및 목표

### 1.1 프로젝트 개요

암호화폐 시장에서 사용자가 설정한 조건에 따라 자동으로 매수/매도를 실행하는 자동 매매 프로그램이다.
사용자는 복잡한 거래 로직을 코드 없이 UI를 통해 설정할 수 있으며, 실시간 차트 분석 기능과
Claude AI 자문 기능을 통해 시장 상황을 모니터링하고 전략을 조정할 수 있다.

### 1.2 핵심 목표

| 목표 | 설명 |
|------|------|
| 자동화 | 24시간 무중단 자동 매수/매도 실행 |
| AI 자문 | Claude AI와 연동하여 매매 전 의견 수렴 |
| 전략 커스터마이징 | 기술적 지표 기반의 매매 조건 설정 |
| 투명성 | 실시간 차트, 거래 내역, 수익률 시각화 |
| 안전성 | 리스크 관리 기능(손절, 익절, 포지션 한도) 내장 |
| 확장성 | 거래소 전환 용이 (테스트넷 → 실거래, 바이낸스 → 원화 거래소) |

### 1.3 타겟 사용자

- 암호화폐 시장에 관심 있는 개인 투자자
- 반복적인 수동 매매를 자동화하고 싶은 트레이더
- 기술적 분석(TA) 기반 전략을 운영하고 싶은 중급 이상 투자자

## 2. 거래소 연동 전략

### 2.1 거래소 선정: 바이낸스 (Binance)

개발 유연성 기준으로 **바이낸스**를 1차 거래소로 확정한다.

| 항목 | 바이낸스 | 업비트 | 빗썸 |
|------|---------|--------|------|
| API 문서 품질 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| WebSocket 지원 | ✅ 완벽 | ✅ 지원 | ⚠️ 제한적 |
| **테스트넷 (Sandbox)** | ✅ 제공 | ❌ 없음 | ❌ 없음 |
| REST API 기능 수 | 매우 많음 | 보통 | 적음 |
| ccxt 지원 품질 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| Rate Limit | 관대함 | 보통 | 엄격함 |
| 거래 수수료 | 0.1% | 0.05% | 0.25% |
| 원화(KRW) 마켓 | ❌ (USDT) | ✅ | ✅ |
| 글로벌 유동성 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |

**바이낸스 선정 근거:**
1. **테스트넷 제공** — 실비용 없이 전체 플로우 검증 가능 (`https://testnet.binance.vision`)
2. **WebSocket 스트림** — 실시간 캔들, 호가, 체결 데이터 완벽 지원
3. **API 기능 완성도** — OCO, 트레일링 스탑 등 고급 주문 타입 서버사이드 지원
4. **ccxt 호환성** — ccxt 라이브러리에서 가장 안정적으로 지원
5. **문서/커뮤니티** — 레퍼런스, 예제 코드가 가장 풍부

### 2.2 거래소 전환 로드맵

```
Phase 1~4 (개발 / 테스트)
    └── 바이낸스 테스트넷
         - 가상 잔고로 전략 검증
         - 실비용 없이 자동매매 로직 전체 테스트

Phase 5 (안정화)
    └── 바이낸스 실거래 (USDT 마켓)
         - 소액으로 실전 검증
         - 환경변수 하나로 테스트넷 ↔ 실거래 전환

Phase 6 (원화 거래소 전환)
    └── 업비트 또는 빗썸 (KRW 마켓)
         - ccxt 추상화 레이어 덕분에 코드 변경 최소화
         - 거래소명만 교체하면 대부분의 로직 재사용 가능
```

### 2.3 거래소 전환 설계 원칙

ccxt 라이브러리를 추상화 레이어로 사용하여 거래소 교체 시 비즈니스 로직 변경을 최소화한다.

```python
# config.py — 환경변수로 거래소 및 환경 제어
# [보안] EXCHANGE_ID 허용리스트로 동적 로드 취약점 방어 (레드팀 지적)
ALLOWED_EXCHANGE_IDS = {"binance", "upbit", "bithumb"}
EXCHANGE_ID   = os.getenv("EXCHANGE_ID", "binance")
if EXCHANGE_ID not in ALLOWED_EXCHANGE_IDS:
    raise ValueError(f"허용되지 않은 거래소: {EXCHANGE_ID}")

# [보안] 문자열 비교 대신 bool 변환으로 안전하게 처리 (레드팀 지적)
USE_TESTNET   = os.getenv("USE_TESTNET", "true").strip().lower() in ("1", "true", "yes")
QUOTE_CURRENCY = os.getenv("QUOTE_CURRENCY", "USDT")  # USDT → KRW 전환 시 변경

# exchange_factory.py
import ccxt

def create_exchange() -> ccxt.Exchange:
    exchange_class = getattr(ccxt, EXCHANGE_ID)   # 위에서 허용리스트 검증 완료
    params = {
        "apiKey": os.getenv("API_KEY"),
        "secret": os.getenv("API_SECRET"),
    }
    if EXCHANGE_ID == "binance" and USE_TESTNET:
        params["urls"] = {
            "api": {
                "public":  "https://testnet.binance.vision/api",
                "private": "https://testnet.binance.vision/api",
            }
        }
    return exchange_class(params)
```

**전환 시 변경 사항:**

| 항목 | 바이낸스 (USDT) | 업비트 (KRW) 전환 시 |
|------|----------------|---------------------|
| `EXCHANGE_ID` | `binance` | `upbit` |
| `QUOTE_CURRENCY` | `USDT` | `KRW` |
| `USE_TESTNET` | `true` → `false` | `false` |
| 비즈니스 로직 | 변경 없음 | 변경 없음 |

## 3. 주요 기능 목록 (기능 명세)

### 3.1 자동 매매 엔진

#### 3.1.1 매수 자동화
- 설정된 매수 조건 충족 시 자동 주문 실행
- 지정가 / 시장가 / 조건부 주문 지원
- 분할 매수 (DCA, Dollar Cost Averaging) 지원
- 매수 수량 설정: 고정 금액, 잔고 비율(%), 고정 수량

#### 3.1.2 매도 자동화
- 설정된 매도 조건 충족 시 자동 주문 실행
- 익절(Take Profit) 자동 실행
- 손절(Stop Loss) 자동 실행
- 분할 매도 (단계별 익절 비율 설정 가능)
- 트레일링 스탑(Trailing Stop) 지원

#### 3.1.3 주문 관리
- 미체결 주문 자동 취소 (타임아웃 설정)
- 주문 재시도 로직 (네트워크 오류 시)
- 주문 체결 실시간 알림

### 3.2 매수/매도 조건 설정

#### 3.2.1 기술적 지표 기반 조건

| 지표 | 매수 조건 예시 | 매도 조건 예시 |
|------|--------------|--------------|
| RSI | RSI < 30 (과매도) | RSI > 70 (과매수) |
| MACD | MACD 골든크로스 | MACD 데드크로스 |
| 이동평균선 (MA) | 단기 MA가 장기 MA 상향 돌파 | 단기 MA가 장기 MA 하향 돌파 |
| 볼린저 밴드 | 하단 밴드 터치 | 상단 밴드 터치 |
| 거래량 | 거래량 급등 (N배 이상) | — |
| 스토캐스틱 | %K < 20 | %K > 80 |
| CCI | CCI < -100 | CCI > 100 |

#### 3.2.2 가격 기반 조건
- 특정 가격 도달 시 매수/매도
- 전일 대비 가격 변동률(%) 조건
- 최고가/최저가 대비 조건

#### 3.2.3 복합 조건 설정
- AND / OR 논리 연산으로 조건 조합
- 조건 그룹 중첩 지원 (예: (조건A AND 조건B) OR 조건C)
- 조건 우선순위 설정

#### 3.2.4 동시 다중 전략 충돌 처리 정책

동일 코인에 대해 여러 전략이 동시에 상반된 신호를 발생시키는 경우 다음 규칙을 따른다.

| 충돌 케이스 | 처리 방식 |
|------------|-----------|
| 매수 신호 + 매도 신호 동시 발생 | **양쪽 모두 스킵**, 사용자에게 충돌 알림 발송 |
| 동일 방향 신호 중복 발생 | 가장 높은 우선순위 전략 1건만 실행 |
| 우선순위 동일 시 | 먼저 조건을 충족한 전략 실행 |

- 전략별 **우선순위(1~10)**를 설정할 수 있으며 숫자가 높을수록 우선 실행
- 충돌 발생 이력은 DB에 저장하여 사용자가 조회 가능
- 충돌 알림은 텔레그램/이메일로 즉시 발송

#### 3.2.5 전략 템플릿
- 사전 정의된 전략 템플릿 제공 (RSI 전략, MACD 전략 등)
- 사용자 정의 전략 저장 및 재사용
- 전략 활성화/비활성화 토글

### 3.3 AI 매매 자문 (Claude API 연동)

매수/매도 주문 실행 **직전**, Claude AI에게 현재 시장 상황과 전략 조건을 전달하여
의견을 받고 실행 여부를 결정한다.

> **[레드팀 보완] AI 자문 구조적 한계 고지**: AI 자문은 전략이 이미 매수 신호를 생성한 후 호출되므로, 프롬프트에 매수 신호 정보가 포함되면 AI가 이를 확증(확증 편향)할 가능성이 높다. 이를 완화하기 위해 ① 프롬프트에 신호 방향을 명시하지 않는 블라인드 모드, ② AI 자문 비활성화 후 전략 단독 운영 A/B 비교 기능을 제공한다. AI 자문은 **보조 수단**으로만 사용하며 최종 책임은 사용자에게 있음을 UI에 명시한다.

#### 3.3.1 자문 모드

| 모드 | 설명 |
|------|------|
| **자동 실행** | AI 의견이 "execute"이면 자동 주문, "avoid"이면 스킵 |
| **반자동 실행** | AI 의견을 알림으로 전달 → 사용자가 최종 승인 |
| **의견만 참고** | 자동 주문은 그대로 진행, AI 의견은 로그/UI에 표시 |

#### `hold` 결정 시 처리 정책

AI decision이 `hold`인 경우 다음 절차를 따른다:

| 단계 | 처리 내용 |
|------|-----------|
| 1 | 주문을 즉시 실행하지 않고 **대기 큐**에 등록 |
| 2 | 설정된 **재평가 주기**(기본값: 5분)마다 조건 재평가 |
| 3 | 재평가 횟수가 **최대 재시도 횟수**(기본값: 3회)를 초과하면 해당 신호 폐기 |
| 4 | 사용자에게 "보류 → 폐기" 알림 발송 |

```
hold 처리 흐름:
대기 큐 등록 → 5분 후 재평가
  ├── 조건 유지 + AI execute → 주문 실행
  ├── 조건 유지 + AI hold   → 재시도 횟수 차감 후 반복
  ├── 조건 유지 + AI avoid  → 신호 폐기
  └── 조건 소멸             → 신호 폐기
```

> 재평가 주기 및 최대 재시도 횟수는 전략별 설정에서 변경 가능하다.

#### 3.3.2 AI 자문 흐름

```
전략 조건 충족
      │
      ▼
Claude API 호출 (claude-opus-4-6 + Adaptive Thinking)
  - 현재 가격 / 24시간 변동률
  - 기술 지표 값 (RSI, MACD, 볼린저밴드 등)
  - 최근 체결 내역
  - 전략 조건 요약
      │
      ▼
AI 응답 수신 (JSON Structured Output)
  {
    "decision": "execute" | "hold" | "avoid",
    "confidence": 0~100,
    "reason": "RSI 과매도 + 거래량 급등으로 반등 가능성...",
    "risk_level": "low" | "medium" | "high",
    "key_concerns": ["BTC 하락 추세 지속 중", ...]
  }
      │
  ┌───┴──────────────────┐
  │                      │
자동실행 모드          반자동 모드
decision=execute       → 사용자 알림 → 승인 시 주문
→ 자동 주문
```

#### 3.3.3 Claude API 구현

```python
import anthropic, json
from typing import Literal

VALID_DECISIONS = {"execute", "hold", "avoid"}
VALID_RISK_LEVELS = {"low", "medium", "high"}

client = anthropic.Anthropic()

async def consult_ai_before_trade(ctx: dict) -> dict:
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        thinking={"type": "adaptive"},
        output_config={
            "format": {
                "type": "json_schema",
                "schema": {
                    "type": "object",
                    "properties": {
                        "decision":     {"type": "string", "enum": ["execute","hold","avoid"]},
                        "confidence":   {"type": "integer"},
                        "reason":       {"type": "string"},
                        "risk_level":   {"type": "string", "enum": ["low","medium","high"]},
                        "key_concerns": {"type": "array", "items": {"type": "string"}}
                    },
                    "required": ["decision","confidence","reason","risk_level"],
                    "additionalProperties": False
                }
            }
        },
        messages=[{"role": "user", "content": build_prompt(ctx)}]
    )
    text = next(b.text for b in response.content if b.type == "text")
    result = json.loads(text)

    # [보안] AI 응답 화이트리스트 검증 — 외부 입력을 신뢰하지 않음 (레드팀 지적)
    if result.get("decision") not in VALID_DECISIONS:
        raise ValueError(f"AI 응답 오염: decision={result.get('decision')}")
    if result.get("risk_level") not in VALID_RISK_LEVELS:
        raise ValueError(f"AI 응답 오염: risk_level={result.get('risk_level')}")
    confidence = result.get("confidence", -1)
    if not isinstance(confidence, int) or not (0 <= confidence <= 100):
        raise ValueError(f"AI 응답 오염: confidence={confidence}")

    return result
```

> **[레드팀 보완] AI 자문 비동기 분리**: AI 자문은 임계 경로(주문 실행 블로킹)가 아닌 **비동기 파이프라인**으로 분리한다. 전략 조건 충족 시 AI 자문 작업을 Celery 비동기 태스크로 제출하고, 결과 수신 후 주문 실행 여부를 결정한다. 타임아웃(5초) 초과 시 자문 스킵 처리 — 단, 자문 스킵 여부는 자문 모드 설정에 따라 달라진다.

```
[비동기 AI 자문 흐름]
전략 조건 충족
      │
      ▼
AI 자문 태스크 제출 (Celery async_apply)
      │
  ┌───┴──────────────────────┐
  │ 5초 이내 응답            │ 5초 초과 타임아웃
  ▼                          ▼
AI 결정 수신             자문 스킵 (설정에 따라)
  │                       의견만 참고 모드: 주문 실행
  ▼                       반자동/자동 모드: 주문 보류 또는 실행
주문 실행 판단
```

#### 3.3.4 백테스트 시 AI 자문 처리

백테스트에서 실제 Claude API를 매 건마다 호출하면 수천 건의 비용이 발생하므로 다음 두 가지 모드를 제공한다.

| 모드 | 방식 | 비용 | 용도 |
|------|------|------|------|
| **경량 모드** | `claude-haiku-4-5` 사용 | 약 1/5 비용 | 빠른 전략 검증 |
| **오프라인 시뮬레이션** | 과거 자문 결과 캐시 재사용 | 무료 | 반복 백테스트 |

- 최초 백테스트 시 AI 자문 결과를 DB에 캐시 저장
- 동일 구간 재테스트 시 캐시를 재사용하여 API 호출 최소화
- 백테스트 설정 화면에서 AI 자문 모드(경량/오프라인/제외) 선택 가능

#### 3.3.5 AI 자문 프롬프트 버전 관리

프롬프트 변경 시 이전 자문 결과와의 일관성이 깨지는 것을 방지하기 위해 버전을 관리한다.

- 프롬프트 변경 시 `prompt_version` 필드를 증가시켜 DB에 함께 저장
- 자문 내역 조회 시 프롬프트 버전별 필터링 지원
- 버전 간 AI 성과 비교 분석 기능 제공

#### 3.3.6 AI 자문 비용 예상

| 항목 | 내용 |
|------|------|
| 모델 | claude-opus-4-6 |
| 입력 비용 | $5 / 1M 토큰 |
| 출력 비용 | $25 / 1M 토큰 |
| 1회 자문 예상 비용 | ~$0.002~0.005 |
| 월 예상 비용 (일 50회 기준) | 약 $3~7 / 월 |
| 최적화 방안 | 소액 주문 자문 스킵, Prompt Caching 활용 |

### 3.8 긴급 정지 기능

서버 장애, 급격한 시장 변동, 사용자 판단에 의한 즉각적인 전체 중단이 필요할 때 사용한다.

| 동작 | 내용 |
|------|------|
| **Redis 분산 락 획득** | `SET emergency_stop_lock NX EX 30` — 동시 요청 레이스 컨디션 방지 |
| 글로벌 잠금 플래그 설정 | Redis에 `global_trading_halt=1` 설정 → 모든 Worker가 신규 주문 차단 |
| 모든 활성 전략 즉시 비활성화 | Celery Worker에 즉시 중단 신호 전송 (revoke + terminate) |
| 미체결 주문 전체 취소 | 거래소 API를 통해 열린 주문 일괄 취소 (부분 실패 시 재시도) |
| 사용자 알림 발송 | 긴급 정지 실행 내역 이메일/텔레그램 통보 |

> **[레드팀 보완] 레이스 컨디션 방지**: 분산 락 없이 동시 긴급 정지 요청이 들어오면 주문 취소가 중복 실행되거나 상태가 불일치할 수 있다. Redis `SET NX` 분산 락으로 단일 실행을 보장한다.

- 대시보드 상단에 **[긴급 정지]** 버튼 상시 노출 (빨간색, 확인 다이얼로그 포함)
- 긴급 정지 해제는 수동으로만 가능 (자동 재개 없음)
- 긴급 정지 이력은 DB에 영구 보관
- 긴급 정지 버튼은 **API 서버 장애 시에도 동작 가능**하도록 Redis 직접 접근 경로(관리용 CLI) 별도 제공

### 3.9 서버 장애 복구

서버 재시작 후 매매 상태를 안전하게 복구하는 절차를 정의한다.

#### 복구 절차

```
서버 재시작
      │
      ▼
미완결 주문 상태 조회 (거래소 API)
      │
  ┌───┴────────────────┐
  │                    │
체결 완료됨          미체결 상태
  │                    │
DB 상태 업데이트     취소 후 재주문 여부 확인
  │                    │
알림 발송         사용자 설정에 따라 처리
                  (자동 재주문 / 취소 / 대기)
```

#### 장애 중 발생 신호 처리 정책

| 상황 | 처리 방식 |
|------|-----------|
| 서버 다운 중 조건 충족 신호 발생 | **소급 실행하지 않음** — 해당 신호 폐기 |
| 재시작 후 조건이 여전히 충족 중 | 새로운 신호로 간주하여 정상 평가 진행 |
| 이유: 과거 시점 가격으로 현재 주문 시 슬리피지 위험 높음 |

### 3.10 계정 및 보안 관리

- API Key 암호화 저장 (AES-256-GCM)
- 사용자 로그인/인증 (이메일 + 2FA)
- **JWT 강제 무효화**: 로그아웃/비밀번호 변경/보안 사고 시 Redis JWT 블랙리스트에 토큰 등록 → 만료 전 토큰 즉시 차단 (레드팀 지적)
- 일일 최대 거래 금액 한도 설정
- **초기 자본금 설정**: 전략 시작 기준 자본 입력 (수익률 계산 기준점)
- **운영 범위 설정**: 단일 사용자(개인) 운영 전제 — 멀티테넌시 미지원 (추후 확장 시 별도 설계 필요)

#### 업비트 비호환 기능 및 대체 방안

| 기능 | 바이낸스 | 업비트 | 대체 방안 |
|------|---------|--------|-----------|
| OCO 주문 | ✅ 서버사이드 | ❌ 미지원 | **서버사이드 대체 구현 필수**: OrderWatcher 서비스가 TP/SL 가격 감시 → 조건 충족 시 시장가 주문 자동 발동 + 반대 주문 취소. 단순 클라이언트 관리 불가 (재시작 시 상태 소실) |
| 트레일링 스탑 | ✅ 서버사이드 | ❌ 미지원 | Celery Worker 서버사이드 가격 추적 → 고점 갱신 시 SL 가격 갱신 → 조건 충족 시 발동 |
| 최소 주문 단위 | $10 (USDT) | 5,000원 (KRW) | 주문 최소 금액 기준값 거래소별 설정으로 분리 |
| 소수점 처리 | 소수점 8자리 | 원 단위 정수 | 수량 계산 시 `math.floor` 처리, 표시 단위 KRW 전환 |

> **[레드팀 보완]** 업비트 OCO는 단순 클라이언트 분리 관리가 아닌 서버사이드 OrderWatcher 서비스로 구현해야 한다. 클라이언트 사이드 구현은 서버 재시작·네트워크 단절 시 미체결 OCO 상태가 소실되어 손절 미작동 위험이 발생한다.

**완료 기준**: 업비트 KRW 마켓에서 자동매매 정상 동작 확인

**총 예상 기간**: 약 21주 (5개월 중반)

## 9. 리스크 및 고려사항

### 9.1 기술적 리스크

| 리스크 | 영향도 | 대응 방안 |
|--------|--------|-----------|

| WebSocket 연결 끊김 | 높음 | 자동 재연결 로직, 하트비트 감시 |

| **Celery Beat SPOF** | **높음** | **Redbeat 스케줄러(Redis 기반) + Redis Sentinel HA 구성** |
| **긴급 정지 레이스 컨디션** | **높음** | **Redis SET NX 분산 락으로 단일 실행 보장** |
| AI 자문 응답 지연 (임계 경로) | 중간 | 비동기 태스크 분리, 타임아웃 5초, 스킵 처리 |

| **업비트 OCO 미지원** | **중간** | **서버사이드 OrderWatcher 구현 필수 (Phase 6)** |
| **슬리피지/수수료 미반영** | 중간 | **수익률 계산에 수수료 + 평균 슬리피지 차감 적용** |
| **백테스트 룩어헤드 바이어스** | 중간 | **현재 Bar 종가 기준만 사용, 미래 데이터 접근 방지 검증** |
| **모니터링 부재** | 중간 | **Prometheus + Grafana 알람 (Phase 5 필수)** |
| 거래소 전환 시 호환성 | 중간 | ccxt 추상화 레이어 활용, Phase 6 회귀 테스트 |
| 서버 장애 중 신호 누락 | 중간 | 재시작 후 소급 실행 않음 — 신호 폐기 후 새 신호 대기 |
| 다중 전략 충돌 | 중간 | 우선순위 기반 처리, 상반 신호 발생 시 양쪽 스킵 |
| **DB 데이터 유실** | 높음 | **PostgreSQL WAL 연속 백업 + 일 1회 스냅샷 (Phase 5 필수)** |

### 9.2 비즈니스/재무 리스크

| 리스크 | 내용 | 대응 방안 |
|--------|------|-----------|
| 전략 오작동 손실 | 조건 설정 오류로 의도치 않은 대량 매매 | 1회 최대 주문 금액 한도 |

## 11. 용어 정리

| 용어 | 설명 |
|------|------|
| RSI | 상대강도지수. 과매수/과매도 판단 지표 (0~100) |
| MACD | 이동평균수렴확산지수. 추세 및 모멘텀 측정 |
| 볼린저 밴드 | 가격의 표준편차로 형성된 상/하단 밴드 |
| DCA | Dollar Cost Averaging. 분할 매수 전략 |
| 트레일링 스탑 | 가격 상승에 따라 손절선을 자동으로 올려주는 기능 |
| MDD | Maximum DrawDown. 최대 낙폭. 리스크 측정 지표 |
| ccxt | 다중 거래소 통합 연동을 지원하는 Python 라이브러리 |
| 테스트넷 | 실제 자산 없이 거래소 API를 테스트할 수 있는 환경 |
| Adaptive Thinking | Claude Opus 4.6의 심층 추론 모드 |
| Structured Output | AI 응답을 JSON 스키마로 강제하는 기능 |
| hold | AI 자문 결과 중 하나. 즉시 실행 보류 후 재평가 대기 상태 |
